**첫 번째로 RCA(Root Cause Analysis)**

전공자라 recheck 자체는 항상 해왔는데, 오늘 강의를 들으면서 "나는 recheck를 제대로 하고 있었나?" 라는 생각이 들었다.

재현 → 관찰 → 원인 후보 → 수정 → 재확인 → 재발 방지, 이 6단계 흐름 자체는 당연히 알고 있었지만...

강사님이 recheck를 안 해서 GitHub Actions가 AWS 리소스를 계속 재생성하다가 **비용 170만원**이 나온 실제 사례를 얘기해주셨는데, 단순히 "됐겠지" 하고 넘어가는 게 얼마나 위험한지 다시 한 번 와닿았다.

더 꼼꼼하게, 같은 기준으로 재확인하는 습관을 더 단단하게 가져가야겠다.

**두 번째로 README를 runbook처럼 쓴다는 개념**

지금까지 README는 "실행되면 OK" 수준으로 썼는데, 오늘 배운 구조는 달랐다.

start → check → expected result → stop → troubleshooting 이 틀로 쓰면 아무것도 모르는 사람이 clone해서 그대로 따라해도 돌아가야 한다는 것.

실행은 됐어도 expected result가 없으면 성공 기준 자체가 없는 거고, stop 절차가 없으면 포트가 계속 점유된다는 것도... 당연한 말인데 내가 쓰던 방식이랑 비교하니까 빠진 게 보였다. 허허

**세 번째, 오늘의 하이라이트! 8교시 챌린지 🎉**

AI로 정적 웹사이트를 직접 만들어보는 시간!

'Promptly'라는 한국 개발자 전용 AI 프롬프트 공유 플랫폼 UI를 만들었다.

**프레임워크 없이** HTML, CSS, JavaScript만 순수하게 써서 만들었는데, CDN도 외부 라이브러리도 backend도 없이 그냥 간단하게\~!

요즘 뭔가 만들면 자동으로 프레임워크 손이 가는데, 순수 JS로 구현하는 거 오랜만이라 나름 재밌었다.

근데 만들다 보니까 **한글 입력 + Enter 키를 누르면 댓글이 두 번 등록되는 버그**가 생겼다.

사실 이 문제... 예전에도 겪었던 적이 있었다. 하지만? 이번 프롬프트 때 새까맣게 잊고 포함하지 못했다 🫪

원인은 한글 IME 조합 특성 때문에 keydown 이벤트가 중복 발생하는 것, 현단계에서는 간단하게 `e.isComposing` 으로 해결했다.

버그를 발견하고, 원인 파악하고, 고치고, 재확인까지 마쳤을 때 뿌듯함도 있었지만 — **앞으로 프롬프트를 작성할 때 이런 edge case를 처음부터 적어두자** 는 교훈이 더 크게 남았다. 알고 있던 문제를 또 만나는 건 내 기록 습관의 문제니까\~?😅

이외에도 Log/설정/비밀값 구분, 관찰 가능성(Log, 상태코드, Metric, Trace), 배포 흐름(build → 산출물 → 릴리스 → 배포 → 되돌리기) 등 내용이 많았는데, 오늘 배운 내용이 다음 주에 배우는 것과 연결된다는 그림이 계속 보여서 방향이 잡히는 느낌이라 좋았다. 👍
