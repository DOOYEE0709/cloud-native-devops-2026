**첫 번째, container 삭제하면 데이터가 사라진다**

오늘 강의 듣기 전에 Day 1에서 volume 없이 만든 PostgreSQL container를 다시 띄워봤는데, 이전에 만든 table이 하나도 없었다.

이유는 SQL 데이터가 image가 아니라 container writable layer에 저장되기 때문이다. container를 `docker rm`으로 삭제하면 writable layer도 같이 사라진다. 같은 image로 새로 만들어도 초기 상태로 시작하는 것.

그래서 volume을 쓰는 거였구나 싶었다. volume을 mount하면 해당 경로는 writable layer를 우회해서 volume에 직접 저장되니까 container를 지워도 데이터가 남는다 🗒️

**두 번째, volume이랑 bind mount**

오늘 mount가 두 종류라는 걸 제대로 이해했다.

named volume은 Docker가 알아서 경로를 관리해주고, container를 삭제해도 데이터가 남는다. PostgreSQL 데이터 저장할 때 쓰는 방식이다.

bind mount는 내 컴퓨터의 실제 폴더를 container에 직접 연결하는 것. host 파일을 수정하면 container에 바로 반영된다. 개발할 때 코드 수정하면서 바로바로 확인하려고 쓰는 방식이라고 한다.

그리고 volume은 container와 독립적으로 존재한다는 게 핵심이었다. container를 지워도 volume은 남아 있다. 근데 `docker volume rm`은 데이터를 영구 삭제하는 거라 조심해야 한다는 것도 배웠다 🗒️

**세 번째, 호스트 접속이랑 내부 container 접속은 다르다**

이게 오늘 제일 헷갈렸던 부분인데, 결국 이렇게 정리됐다.

호스트(내 맥북)에서 DB에 접속할 때는 `-p 15432:5432`로 publish한 포트를 써서 `localhost:15432`로 접속하고,
같은 Docker network 안의 container끼리는 `-p` 없이 container name으로 통신한다.

```bash
# 호스트(내 맥북)에서 접속 — published port 사용
PGPASSWORD=postgres psql -h localhost -p 15432 -U postgres -d postgres -c "SELECT 1;"

# 같은 Docker network의 container에서 접속 — container name 사용
docker run --rm --network paperclip-day2-net -e PGPASSWORD=postgres postgres:16 \
  psql -h paperclip-net-pg -U postgres -d postgres -c "SELECT 1;"
```

container 접속 명령어에서 `-p`를 생략했는데, PostgreSQL 기본 포트가 5432라서 생략하면 자동으로 5432로 접속한다. MySQL이면 기본값이 3306, Redis면 6379 이런 식으로 DB마다 기본 포트가 다르다.

container 안에서 `localhost`를 쓰면 DB가 아니라 client container 자신을 가리킨다는 것도 처음 알았다. 아 아닌가 전공에서 배웠는데, 기억을 못하는건가..? 😅

**네 번째, cleanup — container/volume/network 삭제는 다 다르다**

실습 끝나고 정리할 때 이게 진짜 중요하다고 하셨는데,

| 대상 | 명령 | 없어지는 것 |
|---|---|---|
| container | `docker rm` | process + writable layer |
| volume | `docker volume rm` | 데이터 (복구 불가) |
| network | `docker network rm` | 통신 경로 |

container 지운다고 volume까지 지워지는 게 아니라는 게 포인트. 두 번째 섹션에서 언급했던 것처럼 `docker volume rm`은 DB 데이터를 날리는 명령이라 진짜 조심해야 한다 😱

**다섯 번째, 오토스케일링**

트래픽이 많아지면 서버 한 대로는 감당이 안 되니까, CPU가 50~60%를 넘으면 자동으로 서버를 늘렸다가 트래픽이 줄면 다시 줄인다고 한다.

백엔드, 프론트는 이렇게 오토스케일링이 되는데 DB는 여러 대로 늘리기가 어렵다. 데이터 정합성 문제 때문에 여러 DB가 동시에 같은 데이터를 수정하면 충돌이 생기기 때문이라고 생각한다. 그래서 쓰기는 Primary 하나로 몰고 읽기만 Replica로 분산하는건가?🤔
