## 확인 질문 답변

| 질문 | 답변 |
|---|---|
| 클라우드는 서버가 없는 것인가? | 아니다. 서버는 존재하지만 사용자가 물리 장비를 직접 다루지 않는다. 운영 책임의 일부를 서비스 제공자에게 위임하는 모델이다. |
| 클라우드는 자동으로 싸다는 말이 맞는가? | 규모의 경제로 단가가 낮아질 수 있지만, 사용량/시간/삭제를 관리하지 않으면 비용이 계속 발생한다. 방치 리소스와 잘못된 용량 선택으로 Cloud Cost가 커질 수 있다. |
| 데이터센터 비용은 서버 가격만 보면 되는가? | 아니다. TCO는 장비, 공간, 전력, 냉각, 인력, 장애 위험, 교체 비용까지 포함한다. CAPEX만이 아니라 OPEX와 Risk Cost도 합산해야 한다. |
| AWS가 모든 보안을 책임지는가? | 아니다. 공유 책임 모델에 따라 제공자와 사용자의 책임 경계가 나뉜다. 물리 인프라는 제공자가, 설정/권한/데이터는 사용자가 책임진다. |
| 콘솔에서 만들면 운영이 끝나는가? | 아니다. 권한, 비용, 로그, 삭제, 변경 기록까지 남겨야 한다. 생성보다 운영 책임이 더 중요하다. |
| CAPEX와 OPEX의 차이는 무엇인가? | CAPEX는 서버 구매처럼 먼저 크게 투자하는 비용이고, OPEX는 사용하면서 계속 발생하는 비용이다. 클라우드는 CAPEX를 줄이고 OPEX 성격을 강화한다. |
| Week 1 컴퓨팅 개념과 AWS 서비스는 어떻게 연결되는가? | Compute → EC2/ECS/Lambda, Storage → EBS/S3/RDS, Network → VPC/subnet/security group, Identity → IAM, Observability → CloudWatch 로 연결된다. |

## notes

### 비용 개념 정의

| 개념 | 뜻 | 예시 | 핵심 문장 |
|---|---|---|---|
| CAPEX | 장비나 설비처럼 먼저 투자하는 비용 | 서버 구매, 스토리지 장비, 랙, 네트워크 장비, 구축비 | "먼저 크게 사고 오래 쓴다." |
| OPEX | 운영하면서 계속 발생하는 비용 | 전기, 회선, 운영 인력, 클라우드 월 사용료, 모니터링 | "쓰고 운영하는 동안 계속 낸다." |
| TCO | 전체 소유 비용 | 초기 투자 + 운영비 + 장애/보안/교체 위험 | "진짜 비용은 구매가가 아니라 전체 운영 비용이다." |
| Cloud Cost | 클라우드에서 실제 청구되는 비용 구조 | 사용량 x 단가, 고정 리소스, 데이터 전송, 스토리지, 할인 | "싸질 수 있지만 방치하면 계속 쌓인다." |

```text
TCO = CAPEX + OPEX + Risk Cost
CAPEX = 장비 구매비 + 공간/설비 비용 + 구축 인건비
OPEX = 사용량 비용 + 운영 인건비 + 유지보수 비용
Cloud Cost = 사용량 x 단가 + 고정 리소스 비용 + 데이터 전송/스토리지 비용 - 할인 효과
```

### 데이터센터 운영 책임

| 구성요소 | 직접 운영할 때의 책임 | 운영 리스크 |
|---|---|---|
| Compute | 서버 구매, CPU/RAM 용량 산정, 교체 | 과소 산정, 과대 구매, 노후화 |
| Storage | 디스크/스토리지 장비, 백업, 복구 | 데이터 손실, 용량 부족 |
| Network | 스위치, 라우터, 방화벽, 회선 | 병목, 단절, 보안 구멍 |
| Facility | 전력, 냉각, 랙, 물리 보안 | 정전, 과열, 현장 접근 실패 |
| Operations | 모니터링, 장애 대응, 변경 기록 | 장애 원인 불명, 인수인계 실패 |

### 데이터센터 vs 클라우드 비교

| 기준 | 데이터센터 | 클라우드 |
|---|---|---|
| 소유 | 장비와 공간을 직접 소유하거나 장기 계약 | 자원을 서비스로 사용 |
| 비용 | CAPEX 중심. 초기 투자, 용량 예측, 교체 주기가 중요 | OPEX 중심. 사용량, 단가, 켜진 시간, 삭제 여부가 중요 |
| 속도 | 구매, 배송, 설치 리드타임 존재 | 분 단위 생성 가능 |
| 확장 | 물리 용량 한계와 조달 시간 영향 | 빠른 확장 가능, quota와 비용 제한 존재 |
| 보안 | 물리 접근과 장비 보안까지 직접 관리 | 공유 책임 모델에 따라 설정/권한 책임이 큼 |
| 장애 대응 | 현장 장비와 운영팀 대응 필요 | 제공자 상태와 사용자 설정을 함께 확인 |
| 변경 관리 | 문서/티켓/수작업 변경이 많음 | API/Terraform으로 변경 기록 가능 |

### AWS로 연결

| Week 1 컴퓨팅 개념 | 데이터센터 표현 | AWS에서 만나는 표현 |
|---|---|---|
| Compute | 물리 서버, VM host | EC2, ECS, Lambda |
| Storage | 디스크, NAS, 백업 장비 | EBS, S3, RDS snapshot |
| Network | 스위치, 라우터, 방화벽 | VPC, subnet, route table, security group |
| Identity/access | 출입 통제, 관리자 계정 | IAM user/role/policy |
| Observability | NOC, 로그 서버, 모니터링 | CloudWatch Logs/Metrics/Alarm |
| Handoff | 운영 문서, 변경 티켓 | README, runbook, Terraform plan |

### 핵심 문장

```text
클라우드는 "내 서버가 어딘가에 있다"가 아니라
"운영 책임의 일부를 서비스 제공자에게 넘기고,
남은 책임을 더 명확하게 관리하는 방식"이다.
```

클라우드 학습의 목표는 콘솔 메뉴 암기가 아니라 운영 책임을 질문으로 바꾸는 능력이다.
