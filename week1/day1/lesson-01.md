### 평가 증거 원칙

현업에서 "이해했습니다"는 충분하지 않다. 동료가 다시 확인할 수 있는 기록이 있어야 한다.

| 강한 증거 | 약한 증거 |
|---|---|
| 명령어 출력과 경로 | "해봤다"는 말 |
| 에러 메시지 요약 | "안 됩니다" |
| 스크린샷 파일명 | 화면을 봤다는 진술 |
| README 또는 질문 기록 | 구두 설명만 |

### 첫날 산출물

| 산출물 카드 | 학생이 남길 내용 | 확인 기준 |
|---|---|---|
| 목표 | 5주 후 설명하거나 만들고 싶은 결과 | 관찰 가능한 문장인가 |
| blocker | 현재 걱정되는 부분 또는 `none` | 증상/제약이 구체적인가 |
| 소통 | 공지와 질문 채널 접근 상태 | 다음 질문 경로를 알고 있는가 |
| 안전 | 보호해야 할 정보 목록 | secret/개인정보를 구분하는가 |

### 민감정보 분류

보호해야 할 정보: password, token, API key, MFA 코드, 결제 카드, 주민번호, access key

공개해도 되는 정보: 학습 목표, blocker 증상(비밀값 제외), 작업 폴더 경로, 버전 출력

### AI coding agent 시대 관점

```text
agent는 빠르게 초안을 만들 수 있지만,
목표, 제약, 권한, 검증 기준을 스스로 완성하지 못한다.

좋은 agent 사용자는 "무엇을 만들어줘"보다
"어떤 조건에서 실행되고, 어떤 증거로 성공을 판단할지"를 먼저 말한다.

Day1 목표와 blocker 기록은 이후 agent에게 작업을 지시할 때
필요한 context template이 된다.
```
