## notes

### 재현 가능한 환경에 필요한 기록

| 항목 | 필요한 기록 |
|---|---|
| OS | Windows, macOS, Linux, WSL 여부 |
| runtime | Node.js/Python/Java 버전 |
| package manager | npm, pnpm, pip, poetry |
| dependency | lock file, requirements |
| external program | DB, cache, queue |
| port | 접속 번호 목록 |
| environment variable | `.env.example` |
| initial data | seed data, migration |
| start command | 실행 순서 |
| check command | 정상 확인 방법 |
| stop command | 종료 방법 |

### 같은 환경을 만들기 어렵게 하는 것

```text
설치 링크가 바뀐다.
버전이 자동으로 최신 버전으로 설치된다.
내가 중간에 클릭한 옵션을 기억하지 못한다.
터미널 경로가 다르다.
이미 설치된 프로그램 때문에 결과가 달라진다.
DB 초기 데이터가 다르다.
환경변수 값이 빠진다.
```

"설명 가능한 절차"와 "자동화 가능한 절차"는 다르다. Docker는 이 절차를 더 많이 자동화하게 해 준다.

### README를 실행 계약으로 본다

README는 소개 문서가 아니라 실행 계약이어야 한다. — Day3 lesson-04 Runbook과 같은 맥락.

```text
1. prerequisites  — 필요한 프로그램과 버전
2. install        — 설치 명령
3. configure      — 환경변수, 설정 파일
4. start          — 실행 순서
5. check          — 정상 확인 방법
6. stop           — 종료 방법
7. troubleshoot   — 자주 나는 오류
```

이 순서가 갖춰진 README는 Week2 Docker 수업에서 Dockerfile/Compose로 옮기기 쉽다.

### AI 실험은 재현성이 특히 어렵다

코드 외에도 다음이 모두 기록되어야 한다:

- 모델 버전 — 모델이 바뀌면 결과가 달라진다
- prompt — prompt가 바뀌면 결과가 달라진다
- temperature — 값이 다르면 결과가 달라진다
- embedding model — 바뀌면 검색 결과가 달라진다
- 외부 API 응답 — 시점에 따라 달라질 수 있다

### README의 두 가지 목적

오늘 수업에서 다룬 README는 **실행 계약(Runbook)** 관점이다. 그런데 인사담당자나 채용팀이 보는 README는 목적이 다르다.

| 구분 | 대상 | 중심 내용 |
|---|---|---|
| Runbook README | 팀원, 운영자 | 설치/실행/확인/종료 절차 — "어떻게 실행하는가" |
| 포트폴리오 README | 인사담당자, 면접관 | 기획 배경, 문제 정의, 기술 선택 이유 — "왜 만들었고 무엇을 해결하는가" |

인사담당자가 원하는 README에 들어가면 좋은 것:

```text
1. 프로젝트 한 줄 소개   — 무엇을 만들었는가
2. 기획 배경             — 어떤 문제를 발견했는가, 왜 만들었는가
3. 핵심 기능             — 무엇을 할 수 있는가
4. 기술 스택 선택 이유   — 왜 이 기술을 골랐는가 (단순 나열 말고 이유)
5. 데모 / 스크린샷       — 실제로 어떻게 보이는가
6. 트러블슈팅 경험       — 어떤 문제를 겪었고 어떻게 해결했는가
7. 실행 방법 (Runbook)   — 어떻게 직접 실행해 볼 수 있는가
```

- 기술 스택을 나열만 하면 "쓸 줄 안다"를 증명하지 못한다. 왜 선택했는지, 어떤 트레이드오프가 있었는지를 쓰면 차별화된다.
- 기획 배경이 없으면 "만들어 본 것"과 "문제를 풀기 위해 만든 것"의 차이가 안 보인다.
- Runbook은 포트폴리오 README의 마지막 섹션에 붙이면 두 목적을 모두 충족할 수 있다.

