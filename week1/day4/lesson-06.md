## 확인 질문 답변

| 질문 | 답변 |
|---|---|
| 5단계 이벤트 타임라인을 작성할 수 있는가? | 주문 생성(order created) → 결제 확인(payment confirmed) → 가게 수락(store accepted) → 라이더 배정(rider assigned) → 배달 완료(delivered). 각 단계가 이벤트가 된다. |
| retry 대상 1개를 찾을 수 있는가? | 결제는 성공했지만 가게 알림 이벤트가 실패한 경우, 가게 알림 이벤트를 retry해야 한다. 결제는 이미 성공했으므로 재처리하면 안 된다. |
| 이벤트 처리에서 log가 필요한 이유를 설명할 수 있는가? | 어떤 이벤트가 처리됐는지, 어디서 실패했는지, 언제 재시도됐는지 확인하려면 log가 필요하다. log가 없으면 장애 원인을 파악하기 어렵고 재처리 범위를 알 수 없다. |
| 사용자에게 즉시 보여줄 것과 백그라운드에서 처리할 것을 어떻게 나누는가? | 사용자에게 즉시: 결제 성공 화면, 주문 접수 표시. 백그라운드 처리: 가게 알림, 라이더 배차, 주문 분석. 응답 속도와 안정성의 균형으로 나눈다. |

## notes

### 배달 이벤트 타임라인

| 이벤트 | 사용자에게 필요한 것 | 백그라운드 시스템이 해야 할 것 |
|---|---|---|
| order created | 주문 접수 표시 | 주문 저장 |
| payment confirmed | 결제 성공 표시 | 가게 서비스 알림 |
| store accepted | 조리 상태 표시 | 예상 시간 계산 |
| rider assigned | 배달 진행 표시 | 배차 상태 갱신 |
| delivered | 완료 표시 | 주문 종료와 분석 |

### 실패 시나리오

```text
상황: 결제는 성공했지만 가게 알림 이벤트가 실패했다.

사용자에게 즉시 보여줄 것은? → 결제 성공 화면 (이미 처리됨)
재시도해야 할 이벤트는? → 가게 알림 이벤트
운영자가 먼저 볼 곳은? → queue consumer log, dead letter queue
잃어버리면 안 되는 데이터는? → 결제 정보, 주문 ID
어떤 component의 log가 필요한가? → payment service, order service, notification service
```

### AI 엔지니어링 연결

AI가 배달 서비스에 붙는 경우:
- 배달 시간 예측, 수요 예측
- 이상 주문 탐지, 장애 로그 요약
- 고객 문의 자동 분류

실시간 데이터가 AI 입력이 되면 event stream 품질과 지연시간이 모델 품질과 사용자 경험에 영향을 준다. AI가 추천하거나 예측해도 최종 운영 책임은 로그, 모니터링, rollback, 수동 개입 절차에 남는다.

### Docker 연결

```text
API, queue, worker, log를 함께 실행해야 한다.
→ Docker Compose로 여러 서비스를 함께 실행하게 된다.
  각 서비스가 독립 container에서 실행되고
  network로 연결된다.
```

### 핵심 문장

```text
중요한 것은 모든 일을 한 요청 안에서 끝내는 것이 아니라,
사용자에게 빠르게 알려줄 일과 뒤에서 안정적으로 처리할 일을 나누는 것이다.
```
